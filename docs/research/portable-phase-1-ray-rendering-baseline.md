# Portable Phase 1 ray-rendering baseline

Researched 2026-07-10 for Wayfinder frontier [#3](https://github.com/PetrSeifert/voxel-nexus/issues/3), under the Vulkan 1.3 desktop contract and the five-milestone architecture proof in [#1](https://github.com/PetrSeifert/voxel-nexus/issues/1).

## Recommendation (inference from the sourced facts)

Use a staged progression:

1. Make the first ray Render Path a Vulkan compute pipeline with software voxel traversal, Storage Tier data exposed through storage buffers, and an offscreen storage image as its output.
2. Treat `VK_KHR_ray_query` inside that compute pipeline as a later optional capability tier if measurements show that an acceleration structure helps.
3. Add `VK_KHR_ray_tracing_pipeline` as a separate optional Render Path only if its distinct shader scheduling and traversal model produces a demonstrated benefit.

Do not make hardware ray tracing plus a compute fallback one Phase 1 deliverable. Vulkan supplies optional capabilities, not an automatic fallback implementation: supporting both immediately would require the engine to own and validate two traversal/data-preparation paths before the architecture proof needs them.

This recommendation preserves the repository's portable Vulkan 1.3 core contract and keeps candidate 4 focused on proving that a second Render Path can consume the same Voxel Frontend. It does not claim that software traversal will be faster than an acceleration-structure path. Performance and representation choices such as one AABB per occupied voxel, triangles for exposed faces, or coarser procedural bounds need a later measured prototype.

## Sourced facts

### Core compute fits the portability floor

Compute pipelines and `vkCmdDispatch` are Vulkan 1.0 core functionality. If an implementation exposes graphics operations, it must expose at least one queue family on at least one physical device that supports both graphics and compute operations. The Render Backend should still select and record against a queue family advertising `VK_QUEUE_COMPUTE_BIT`; queue-family grouping is implementation-dependent. [Vulkan pipeline specification](https://docs.vulkan.org/spec/latest/chapters/pipelines.html#pipelines-compute) · [Vulkan queue flags](https://docs.vulkan.org/refpages/latest/refpages/source/VkQueueFlagBits.html)

Storage buffers and storage images are core descriptor types. A compute shader can read or write storage buffers, and can store to a storage image when the image format advertises `VK_FORMAT_FEATURE_STORAGE_IMAGE_BIT`. Buffer and image capabilities are finite and queryable: the Vulkan core minimum for `maxStorageBufferRange` is 2^27 bytes, while the core minimum for `maxImageDimension3D` is only 256. [Vulkan resource descriptors](https://docs.vulkan.org/spec/latest/chapters/descriptors.html) · [Vulkan limit requirements](https://docs.vulkan.org/spec/latest/chapters/limits.html#limits-minmax) · [Vulkan format support](https://docs.vulkan.org/spec/latest/chapters/formats.html#formats-properties)

Consequently, a portable data contract cannot assume that an entire large Voxel Volume fits in one storage-buffer binding or one 3D image. The Render Path must query actual limits and partition or page its shader-visible data when necessary. This is an inference from the required minimums, not a mandated representation.

Vulkan 1.3 requires support for buffer device addresses, so a later pointer-based sparse representation can use core `VkDeviceAddress` functionality after enabling the feature. The compute baseline does not need physical addresses: ordinary storage-buffer descriptors are sufficient. General 64-bit integer arithmetic and byte-sized storage are separate features; `shaderInt64` must be supported and enabled before shaders use 64-bit integer types, and `storageBuffer8BitAccess` remains optional in Vulkan 1.3. A strictly portable first representation should therefore use 32-bit indexing and either 32-bit voxel words or explicitly query and enable byte-storage support. [Buffer-device-address promotion](https://docs.vulkan.org/refpages/latest/refpages/source/VK_KHR_buffer_device_address.html#_promotion_to_vulkan_1_3) · [`VkPhysicalDeviceFeatures`](https://docs.vulkan.org/refpages/latest/refpages/source/VkPhysicalDeviceFeatures.html) · [8-bit-storage promotion](https://docs.vulkan.org/refpages/latest/refpages/source/VK_KHR_8bit_storage.html#_promotion_to_vulkan_1_2)

Writing directly to a swapchain image as a storage image is not a portable assumption. Surface capabilities report the allowed swapchain `imageUsage` flags, and only `VK_IMAGE_USAGE_COLOR_ATTACHMENT_BIT` is required. The portable shape is therefore an offscreen storage image followed by a Render Backend presentation step; that step may use a copy when the surface advertises the required usage, or a graphics composite into the guaranteed color attachment. [Vulkan WSI surface capabilities](https://docs.vulkan.org/spec/latest/chapters/VK_KHR_surface/wsi.html#_surface_capabilities) · [`VkSwapchainCreateInfoKHR` valid usage](https://docs.vulkan.org/spec/latest/chapters/VK_KHR_surface/wsi.html#VkSwapchainCreateInfoKHR)

### KHR ray traversal is an optional capability family

Cross-vendor Vulkan ray tracing is exposed by interrelated extensions rather than by the Vulkan 1.3 core contract. On Vulkan 1.3, `VK_KHR_acceleration_structure` still depends on `VK_KHR_deferred_host_operations`; `VK_KHR_ray_query` and `VK_KHR_ray_tracing_pipeline` each depend on `VK_KHR_acceleration_structure`. An application must enumerate the device extensions, query their feature structures, and enable the required feature bits (`accelerationStructure` and either `rayQuery` or `rayTracingPipeline`) before use. [Khronos ray-tracing guide](https://docs.vulkan.org/guide/latest/extensions/ray_tracing.html) · [`VK_KHR_acceleration_structure` dependencies](https://docs.vulkan.org/refpages/latest/refpages/source/VK_KHR_acceleration_structure.html) · [`VK_KHR_ray_query` dependencies](https://docs.vulkan.org/refpages/latest/refpages/source/VK_KHR_ray_query.html) · [`VK_KHR_ray_tracing_pipeline` dependencies](https://docs.vulkan.org/refpages/latest/refpages/source/VK_KHR_ray_tracing_pipeline.html) · [Vulkan feature queries and enabling](https://docs.vulkan.org/spec/latest/chapters/features.html)

Both optional paths require application-managed acceleration structures. The application queries build sizes, allocates acceleration-structure and scratch buffers, builds or updates the structures, and synchronizes build and traversal access. KHR acceleration-structure geometry is supplied as triangles or AABBs and assembled into opaque bottom-level and top-level structures; it does not traverse an arbitrary Storage Tier directly. [Khronos ray-tracing guide](https://docs.vulkan.org/guide/latest/extensions/ray_tracing.html#_vk_khr_acceleration_structure) · [Vulkan acceleration structures](https://docs.vulkan.org/spec/latest/chapters/accelstructures.html)

Acceleration-structure updates are deliberately constrained. An update cannot change geometry or instance counts, primitive counts, or active/inactive state; those changes require a new build. Therefore, mapping occupancy directly to acceleration-structure primitives can turn voxel creation or removal into a rebuild. The size and frequency of that cost depend on the eventual mapping and workload and cannot be decided from the specification alone. [Acceleration-structure update rules](https://docs.vulkan.org/spec/latest/chapters/accelstructures.html#acceleration-structure-update)

`VK_KHR_ray_query` runs traversal inline in existing shader stages, including compute. It adds no API commands and does not launch separate hit or miss shaders, but it still consumes an acceleration structure. For procedural AABB geometry, the query returns a candidate to the calling shader, which must test the desired geometry and generate a confirmed intersection. Khronos notes that inline traversal offers implementations fewer scheduling and tracing optimization opportunities than a ray-tracing pipeline. [Ray-query extension](https://docs.vulkan.org/refpages/latest/refpages/source/VK_KHR_ray_query.html) · [Ray-query guide](https://docs.vulkan.org/guide/latest/extensions/ray_tracing.html#_vk_khr_ray_query) · [AABB ray-query confirmation](https://docs.vulkan.org/spec/latest/chapters/raytraversal.html#_aabb_intersection_candidates)

`VK_KHR_ray_tracing_pipeline` is a pipeline type separate from compute and graphics. It adds ray-generation, intersection, any-hit, closest-hit, miss, and callable shader stages, shader groups and a shader binding table, and dedicated trace commands. This is a materially different execution and resource model from the compute Render Path. [Ray-tracing pipeline extension](https://docs.vulkan.org/refpages/latest/refpages/source/VK_KHR_ray_tracing_pipeline.html) · [Vulkan ray-tracing pipeline execution](https://docs.vulkan.org/spec/latest/chapters/raytracing.html)

## Capability and data-access contract

| Tier | Required capability | Voxel data access | Output | Additional owned state |
| --- | --- | --- | --- | --- |
| Portable baseline | Vulkan 1.3; a queue with `VK_QUEUE_COMPUTE_BIT`; queried buffer limits and storage-image format support | Descriptor-backed storage buffers using 32-bit indexing by default | Offscreen storage image, then capability-aware presentation | Compute pipeline, descriptors, software traversal data |
| Optional ray query | Baseline plus `VK_KHR_deferred_host_operations`, `VK_KHR_acceleration_structure`, `VK_KHR_ray_query`, and their feature bits | Acceleration structure for coarse geometry plus descriptors/device addresses for voxel and material payloads | Same compute/storage-image shape | BLAS/TLAS, build inputs, scratch memory, build/update synchronization |
| Optional ray pipeline | Acceleration-structure requirements plus `VK_KHR_ray_tracing_pipeline` and its feature bit | Acceleration structure plus shader-visible payload buffers | Storage image or another image consumed by presentation | Separate pipeline, shader groups, shader binding table, pipeline-specific limits and synchronization |

The seam shared across tiers should be semantic rather than an acceleration-structure layout: camera/ray inputs, Voxel Scene snapshot or view, voxel/material identifiers, hit results, and image output. The exact GPU representation remains owned by each Render Path so that adding the optional tiers does not make either the Voxel Frontend or a Storage Tier depend on KHR acceleration structures. This seam is an architectural inference consistent with the repository glossary; its precise ownership and update protocol remain for the separate interface frontier.

## Roadmap consequence (inference)

Keep the current candidate 4 direction: demonstrate a portable compute ray Render Path alongside rasterization. Do not add hardware ray tracing to the first five-milestone acceptance contract. Record it as a post-proof option, with ray query investigated before a full ray pipeline because it preserves the existing compute dispatch, shading, and output shape while isolating the acceleration-structure experiment.
